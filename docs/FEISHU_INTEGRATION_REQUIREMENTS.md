## 1. 背景与目标

### 1.1 背景

CrabCoder 是一款纯 Go 实现的 AI 编程助手，核心能力包括任务分解、并发 DAG 执行、多模型支持及安全管控。目前交互方式为 CLI 终端和未来的 IDE 插件。

为拓展使用场景，方便团队成员在飞书中直接与 CrabCoder 交互，需要将飞书作为新的前端入口，实现以下场景：
- 用户通过飞书向 CrabCoder 发送自然语言指令
- CrabCoder 执行任务后将结果反馈到飞书会话

### 1.2 目标

| 目标 | 说明 |
|------|------|
| 消息通道 | 实现飞书机器人接收用户命令，透传给 CrabCoder 引擎 |
| 流式反馈 | 支持飞书消息流式反馈（任务开始、进展、需要确认、最终结果） |
| 能力不变 | 保持 CrabCoder 原有的任务分解、并发调度、安全审批等核心能力 |
| 部署简便 | 支持本地运行+内网穿透或企业自建应用 |

### 1.3 非目标

- 不替代 CrabCoder 现有的 CLI/IDE 前端，而是增加一个通道
- 不要求飞书作为唯一控制端，多通道可共存

## 2. 用户角色与场景

### 2.1 用户角色

| 角色 | 说明 | 典型场景 |
|------|------|---------|
| 开发者 | 使用 CrabCoder 完成编程任务 | 在飞书群聊中 @ 机器人，要求"重构 auth 模块，添加双因素认证" |
| 审批者 | 需要批准高风险操作 | 机器人发送卡片请求确认，点击"允许"或"拒绝" |
| 管理员 | 配置机器人、查看审计日志 | 设置机器人命令权限、查看任务执行历史 |

### 2.2 典型场景

1. **一对一私聊**：用户直接向机器人发消息，机器人回复执行结果
2. **群聊协作**：在项目群中 @ 机器人，机器人执行后回复到群或私信发送详细结果

## 3. 功能需求

### 3.1 飞书机器人基础能力

| 需求ID | 功能 | 描述 |
|--------|------|------|
| FB-01 | 接收消息 | 机器人能够接收用户 @ 消息或私聊文本，支持飞书开放平台的事件订阅（HTTP Webhook 或 WebSocket 长连接） |
| FB-02 | 发送普通文本消息 | 能够向用户或群聊发送文本结果 |
| FB-03 | 发送富文本/卡片消息 | 支持发送飞书消息卡片（interactive card），用于任务进度展示、风险确认、多选项交互 |
| FB-04 | 消息延迟处理 | 对于长时间执行的任务，机器人应先回复"任务已接收，正在处理..."，避免飞书超时 |
| FB-05 | 流式进度推送 | 当 CrabCoder 内部有子任务状态变化时，机器人可以向用户推送进度消息 |

### 3.2 任务接收与解析

| 需求ID | 功能 | 描述 |
|--------|------|------|
| FB-10 | 命令提取 | 从飞书消息中提取用户意图（去除 @ 机器人部分），作为 CrabCoder Engine 的输入 |
| FB-11 | 上下文传递 | 支持用户附带当前文件路径或项目信息，供 CrabCoder 理解上下文 |
| FB-12 | 多轮对话支持 | 保留飞书会话 ID 与 CrabCoder 会话 ID 的映射，支持连续对话 |

### 3.3 任务执行与干预

| 需求ID | 功能 | 描述 |
|--------|------|------|
| FB-20 | 执行引擎 | 将用户消息转发给 `Engine.ProcessRequest`，获得异步事件流 |
| FB-21 | 风险操作确认 | 当遇到高风险任务时，机器人发送确认卡片，用户点击按钮后回传决策给引擎 |
| FB-22 | 任务取消 | 用户可通过发送 `/cancel` 或点击卡片上的"取消"按钮终止任务 |
| FB-23 | 任务重试 | 对于失败的任务，用户可发送 `/retry` 指令重新执行 |

### 3.4 结果反馈

| 需求ID | 功能 | 描述 |
|--------|------|------|
| FB-30 | 最终结果汇总 | 完成所有子任务后，将汇总结果（自然语言 + 文件变更摘要）通过飞书消息发送 |
| FB-31 | 文件变更展示 | 如果任务涉及文件创建/修改，应在消息中列出变更的文件列表 |
| FB-32 | 错误处理 | 若任务分解失败、DAG 循环依赖或执行异常，返回友好的错误提示 |

### 3.5 安全与权限

| 需求ID | 功能 | 描述 |
|--------|------|------|
| FB-40 | 用户认证 | 信任飞书提供的用户身份（通过事件中的 `open_id` 或 `user_id`），无需额外登录 |
| FB-41 | 操作审计 | 记录每条飞书指令、执行结果、确认动作，输出到审计日志 |
| FB-42 | 工作空间隔离 | 不同用户或群聊的任务应在独立的项目工作区执行（可配置） |

---

## 4. 非功能需求

| 类别 | 要求 |
|------|------|
| 响应时间 | 飞书消息接收后 3 秒内必须回复"已收到"或开始处理；最终结果根据任务复杂度可长可短，但超时需主动告知 |
| 并发处理 | 支持多个飞书会话同时调用 CrabCoder 引擎，引擎内部通过 worker pool 控制并发 |
| 可靠性 | 若 CrabCoder 进程重启，已收到的消息未完成的任务应能恢复或通知用户重新发起 |
| 部署简便 | 提供 Docker 镜像或单一二进制，支持通过配置文件指定飞书 App ID/Secret 和回调地址 |
| 可观测性 | 暴露 Prometheus 指标（接收消息数、任务执行时长、失败率等） |

## 5. 接口与集成设计

### 5.1 组件交互

```
┌─────────────┐     ┌─────────────────┐     ┌─────────────────┐     ┌─────────────┐
│  飞书用户    │────▶│  飞书开放平台    │────▶│  飞书适配器(Go) │────▶│ CrabCoder  │
└─────────────┘     └─────────────────┘     └─────────────────┘     └─────────────┘
     │                                            │                       │
     │                                            │                       │
     ◀────────────────────────────────────────────┴───────────────────────
                          消息卡片/文本/确认回调
```

### 5.2 飞书适配器模块职责

| 模块 | 职责 |
|------|------|
| 事件网关 | 接收飞书的事件回调，验证签名，解析出消息内容、用户 ID、会话 ID |
| 会话管理器 | 维护飞书 `session_id` ↔ CrabCoder `request_id` 的映射，支持中断续传 |
| 消息发送器 | 调用飞书 API 发送文本、卡片、文件 |
| 确认处理器 | 接收飞书卡片按钮点击事件，转换为 `engine.ConfirmRequest` 调用 |

### 5.3 数据结构扩展

```go
// pkg/model/feishu.go
type FeishuContext struct {
    Platform     string // "feishu"
    SessionID    string // 飞书会话 ID
    UserID       string // 飞书 open_id
    MessageID    string // 飞书消息 ID
    ResponseType string // "direct" 或 "group"
}

// engine.Request 结构体增加 PlatformContext 字段
type Request struct {
    Text            string
    Mode            string // "ask" 或 "chat"
    SessionID       string
    PlatformContext interface{} // 存储 FeishuContext
}
```

### 5.4 飞书事件订阅配置

| 配置项 | 值 |
|--------|-----|
| 事件类型 | `im.message.receive_v1`（接收消息） |
| 卡片交互回调 | `card.action.trigger`（按钮点击） |
| 接入方式 | 推荐使用 WebSocket 模式（无需公网 IP），或 HTTP + 内网穿透（如 ngrok） |

## 6. 部署与运维

### 6.1 部署架构

```
┌──────────────────────────────────────────────────────────────┐
│                        飞书客户端                             │
└────────────────────────────┬─────────────────────────────────┘
                             │
                             ▼
┌──────────────────────────────────────────────────────────────┐
│                    飞书开放平台（事件订阅）                    │
└────────────────────────────┬─────────────────────────────────┘
                             │ WebSocket/HTTPS
                             ▼
┌──────────────────────────────────────────────────────────────┐
│  ┌─────────────────┐      ┌─────────────────┐                │
│  │   飞书适配器模块  │ ───▶ │  CrabCoder Engine │                │
│  └─────────────────┘      └─────────────────┘                │
└──────────────────────────────────────────────────────────────┘
```

### 6.2 配置项

```yaml
# config.yaml
feishu:
  enabled: true
  app_id: "cli_xxxxx"
  app_secret: "xxxxx"
  # 使用 WebSocket 模式（推荐）
  event_mode: "websocket"   # 或 "http"
  http_callback_url: "https://your-domain.com/webhook"  # 仅 http 模式需要
  verification_token: "xxxxx"
  # 消息发送配置
  message_timeout: 30s
  max_retries: 3
```

### 6.3 启动命令

```bash
crabcoder --config config.yaml --enable-feishu
```

## 7. 验收标准

| # | 验收项 | 预期行为 |
|---|--------|---------|
| 1 | 基础交互 | 用户发送"你好"，机器人回复"你好，我是 CrabCoder，请问有什么可以帮您？" |
| 2 | 任务执行 | 用户发送"列出当前目录文件"，机器人执行 `ls`（需确认后执行），返回文件列表 |
| 3 | 任务分解 | 用户发送"重构 main.go 中的函数"，机器人展示任务分解进度，最终返回修改结果 |
| 4 | 风险确认 | 高风险命令（如 `rm -rf`）触发确认卡片，用户点击"允许"后执行，"拒绝"后取消 |
| 5 | 并发隔离 | 多用户同时使用，无消息串扰 |
| 6 | 审计日志 | 所有交互记录出现在审计日志中 |
